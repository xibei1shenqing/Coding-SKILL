# 公共前置规则：需求明确性检查

## 适用范围

本规则适用于所有代码方案类 reference，包括但不限于：

- 新项目：从 0 到 1 创建项目
- 老项目：添加功能
- 老项目：修改功能
- 老项目：修复 Bug
- 老项目：重构优化

在任何方案文件创建之前，都必须先判断用户目标是否明确。

------

## 核心原则

AI 不应在用户目标不明确时直接生成方案文件。

必须先确认：

1. 用户到底要做什么
2. 用户期望得到什么结果
3. 当前问题或需求的边界是什么
4. 哪些内容不属于本次处理范围
5. 是否能判断应该进入哪个 reference

如果这些信息不足，应先询问用户，而不是猜测后直接生成方案。

------

## 判断标准

### 1. 添加功能是否明确

当用户要添加功能时，至少需要明确：

- 要新增什么功能
- 这个功能由谁使用
- 入口在哪里
- 输入是什么
- 输出是什么
- 成功后应产生什么结果
- 本次不做什么

如果用户只说：

```
帮我加个导出功能
帮我加个通知功能
帮我加个后台筛选
帮我加个任务暂停
```

但没有说明功能边界，应先询问。

推荐询问：

```
在创建添加功能方案前，我需要先确认以下信息：

1. 你要新增的功能具体是什么？
2. 用户从哪里触发这个功能？
3. 这个功能需要接收哪些输入？
4. 成功后应该产生什么输出或结果？
5. 哪些情况本次明确不需要处理？
```

------

### 2. 修改功能是否明确

当用户要修改功能时，至少需要明确：

- 要修改哪个已有功能
- 旧行为是什么
- 新行为希望变成什么
- 新旧行为的主要差异
- 是否需要兼容旧行为
- 是否涉及历史数据

如果用户只说：

```
帮我改一下登录逻辑
帮我改一下任务状态
帮我把这个功能优化一下
帮我改成多课程
```

但没有说明旧行为和新行为，应先询问。

推荐询问：

```
在创建修改功能方案前，我需要先确认以下信息：

1. 要修改的是哪个已有功能？
2. 当前旧行为是什么？
3. 你希望修改后的新行为是什么？
4. 哪些旧行为需要继续兼容？
5. 是否涉及已有数据或历史记录？
```

------

### 3. 修复 Bug 是否明确

当用户要修复 Bug 时，至少需要明确：

- Bug 现象是什么
- 如何复现
- 期望结果是什么
- 实际结果是什么
- 是否有报错日志
- 程序是无法运行，还是运行后结果不对

如果用户只说：

```
这个接口有 bug
页面不正常
任务状态不对
程序跑不起来
这里报错了
```

但没有说明现象、复现步骤、期望结果和实际结果，应先询问。

推荐询问：

```
在创建 Bug 修复方案前，我需要先确认以下信息：

1. Bug 的具体现象是什么？
2. 复现步骤是什么？
3. 期望结果是什么？
4. 实际结果是什么？
5. 是否有报错日志或截图？
6. 这是程序无法运行，还是功能结果不符合预期？
```

------

### 4. 重构优化是否明确

当用户要重构优化时，至少需要明确：

- 要优化哪段代码或哪个模块
- 优化目标是什么
- 是否允许改变功能行为
- 是否允许调整文件结构
- 是否允许引入新依赖
- 验收标准是什么

如果用户只说：

```
帮我优化一下
帮我重构一下
这段代码太乱了
帮我提升性能
```

但没有说明优化目标，应先询问。

推荐询问：

```
在创建重构优化方案前，我需要先确认以下信息：

1. 要重构或优化的是哪个文件、模块或功能？
2. 主要目标是提升可读性、性能、可维护性，还是减少重复代码？
3. 是否必须保持功能行为完全不变？
4. 是否允许调整文件结构？
5. 如何判断优化完成？
```

------

## 信息不足时的处理方式

如果需求不明确，不要创建方案文件。

只输出需要用户补充的信息。

回复格式：

```
当前还不能创建方案文件，因为需求目标不够明确。

我需要你先补充：

1. ...
2. ...
3. ...

补充后我再进入对应 reference，读取项目上下文并创建方案文件。
```

------

## 信息足够时的处理方式

如果需求已经明确，可以进入对应 reference 的后续流程。

回复格式：

```
需求目标已经明确：

- 任务类型：
- 目标：
- 期望结果：
- 本次边界：

下面进入对应 reference，继续读取项目上下文并创建方案文件。
```

------

## 不要过度追问

只询问创建方案所必需的信息。

不要一次性问过多细节。

如果可以通过阅读项目文件获得的信息，不要优先询问用户，应在进入项目上下文阅读阶段通过项目内容确认。

例如：

- 项目技术栈
- 文件路径
- 已有函数名称
- 数据模型字段
- 路由位置
- 配置文件位置

这些通常应该通过读项目获得，而不是先问用户。

------

## 关键边界

需求澄清阶段只负责确认“用户想要什么”。

不要在这个阶段：

- 设计技术方案
- 输出文件变更计划
- 输出函数设计
- 输出数据结构
- 编写代码
- 创建方案文件
- 猜测项目实现

只有用户目标明确后，才允许进入项目上下文阅读和方案文件创建阶段。
