# 1.0 新项目：从 0 到 1 创建项目方案 Reference

## 适用场景

当用户希望从零开始创建一个新项目，且当前不存在可继续开发的完整项目代码时，使用本 reference。

典型场景：

- 用户要从 0 到 1 创建一个新系统
- 用户只有一个项目想法，还没有项目结构
- 用户只有零散需求，还没有完整方案
- 用户想先确定项目目标、MVP、技术栈、数据模型、模块划分和开发顺序
- 用户想创建一个新后端项目、新前端项目、全栈项目、自动化脚本项目、工具项目或服务项目

核心要求：

- 当前重点是创建项目级方案
- 当前不是在已有项目中添加功能
- 当前不是修改已有功能
- 当前不是修复已有 Bug
- 当前不是重构优化已有代码
- 当前不直接写代码，只创建项目方案文件

本 reference 只负责 **新项目创建前的项目级方案设计**，不负责代码实现。

------

## 不适用场景与转入规则

如果在需求明确性检查后，发现用户需求不属于“新项目”，必须停止使用本 reference，并转入对应 reference。

转入规则：

- 如果用户是在已有项目中新增一个原本不存在的功能，应转入本 skill 根目录的 `references/old_project_add_feature.md`
- 如果用户是在已有项目中修改已有功能行为，应转入本 skill 根目录的 `references/old_project_modify_feature.md`
- 如果用户是在已有项目中修复错误、异常或不符合预期的行为，应转入本 skill 根目录的 `references/old_project_fix_bug.md`
- 如果用户是在已有项目中重构优化代码，且功能行为不变，应转入本 skill 根目录的 `references/old_project_refactor_optimize.md`

特别注意：

新项目方案必须满足：

```
当前没有可作为基础继续开发的完整项目，或者用户明确希望重新从 0 到 1 规划项目。
```

如果用户已经有现成项目，并且只是想在其中继续开发，不应继续使用本 reference。

判断后不要继续创建新项目方案文件。

应先在聊天中说明转入原因，例如：

```
当前需求不属于从 0 到 1 创建新项目，因为用户已经有现成项目，并且目标是在旧项目中新增功能。
因此应转入 references/old_project_add_feature.md。
```

------

## 核心目标

本 reference 的目标不是在聊天中输出完整方案，而是在项目中创建或更新一个独立的 Markdown 新项目方案文件。

该方案文件用于：

- 明确项目目标
- 明确用户角色和使用场景
- 明确 MVP 范围
- 明确本次不做什么
- 明确核心业务闭环
- 明确输入输出
- 明确数据模型
- 明确模块划分
- 明确技术栈
- 明确项目目录结构
- 明确核心接口、页面或命令
- 明确第一版开发顺序
- 明确验收标准
- 明确风险和边界
- 避免多轮聊天污染上下文
- 让后续代码实现流程只读取方案文件
- 保持“项目方案设计”和“代码实现”分离

新项目方案重点是 **项目级规划**，不是单个功能的详细实现设计。

------

## 重要限制

使用本 reference 时，AI 不允许直接写代码。

禁止行为：

- 不要输出完整代码实现
- 不要生成项目代码
- 不要生成 patch
- 不要直接创建业务文件代码
- 不要把项目方案和代码实现混在一起
- 不要在聊天中反复输出整份方案
- 不要跳过需求明确性检查
- 不要一开始就设计过度复杂架构
- 不要把 MVP 扩大成完整商业系统
- 不要在需求不明确时直接创建方案文件
- 不要把单个功能方案写成项目方案
- 不要为小项目引入不必要的复杂依赖
- 不要默认用户需要后台、权限、队列、微服务、分布式架构，除非需求需要

如果用户要求“直接写代码”或“直接创建项目”，也应先创建或更新新项目方案文件，并说明：

```
当前阶段先创建新项目方案文件，代码实现应在方案确认后由代码实现流程处理。
```

------

## 当前重点

本 reference 关注的是：

> 一个新项目如何从 0 到 1 成立。

在方案设计过程中，重点需要回答以下问题：

- 这个项目为什么要做
- 这个项目给谁用
- 用户在什么场景下使用
- 第一版必须完成什么
- 第一版明确不做什么
- 项目的最小可用闭环是什么
- 系统接收什么输入，输出什么结果
- 需要哪些核心数据模型
- 需要哪些核心模块
- 技术栈如何选择
- 项目目录如何组织
- 第一版按什么顺序开发
- 做到什么程度算完成
- 主要风险和边界是什么

所有方案设计都应围绕上述问题展开。

------

## 工作流程

### 第零步：需求明确性检查

在执行本 reference 前，必须先读取并执行本 skill 根目录的 `references/common_requirement_clarification.md`，完成需求明确性检查。

只有当新项目目标足够明确后，才允许继续本 reference 的后续流程。

新项目至少需要明确：

- 项目要解决什么问题
- 项目大概给谁使用
- 第一版希望完成什么
- 用户期望项目产生什么结果
- 是否有指定技术栈
- 是否有明确不做的内容
- 是否是新项目，而不是已有项目继续开发

如果以上信息不足，不要创建方案文件、不要输出完整方案，应先向用户询问最小必要信息。

------

### 第一步：收集新项目背景

在创建方案文件前，必须先收集与新项目相关的背景信息。

需要尽可能确认：

- 项目目标
- 用户角色
- 使用场景
- 核心业务流程
- MVP 范围
- 非目标范围
- 输入数据
- 输出结果
- 数据来源
- 是否需要数据库
- 是否需要前端页面
- 是否需要后端接口
- 是否需要命令行或脚本
- 是否需要定时任务
- 是否需要异步任务
- 是否需要第三方服务
- 是否需要用户登录和权限
- 是否有指定技术栈
- 是否有部署环境要求
- 是否有性能、安全、成本、维护要求

如果用户已经提供文档、草稿、需求说明或已有空项目结构，应优先阅读这些内容。

不要在缺少核心目标和 MVP 范围的情况下直接设计项目。

------

### 第二步：判断信息是否足够

如果信息不足，应要求用户提供必要内容。

回复格式：

```
在创建新项目方案前，我还需要确认以下信息：

1. ...
2. ...
3. ...

请补充这些信息后再继续。
```

如果信息足够，应先简要说明当前理解，然后创建方案文件。

回复格式：

```
当前对新项目的理解：

- 项目目标：
- 使用对象：
- 核心场景：
- 第一版范围：
- 明确不做：

下面将创建新项目方案文件。
```

------

### 第三步：创建方案文件

AI 必须根据 `assets/new_project_plan_template.md` 创建独立方案文件。

推荐存放路径：

```
docs/ai-plans/new-project/<project_name>.md
```

如果项目中没有该目录，应创建：

```
docs/ai-plans/new-project/
```

文件命名规则：

```
<项目名称的英文或拼音简写>.md
```

示例：

```
docs/ai-plans/new-project/task_platform.md
docs/ai-plans/new-project/course_automation.md
docs/ai-plans/new-project/inventory_system.md
docs/ai-plans/new-project/data_cleaning_tool.md
```

创建方案文件时：

- 必须使用 `assets/new_project_plan_template.md`
- 必须填充模板内容
- 必须基于用户需求和已收集背景填写
- 必须明确 MVP 范围
- 必须明确本次不做什么
- 必须明确项目级模块划分
- 必须明确第一版开发顺序
- 不要在聊天中完整复制方案正文
- 不要写代码

------

### 第四步：创建完成后的聊天回复

创建方案文件后，聊天中只需要简要说明文件位置和内容概要。

回复格式：

```
已创建新项目方案文件：

docs/ai-plans/new-project/<project_name>.md

该文件包含：

1. 项目概述
2. 用户角色和使用场景
3. MVP 范围
4. 非目标范围
5. 核心业务闭环
6. 输入输出
7. 数据模型草案
8. 模块划分
9. 技术栈方案
10. 项目目录结构
11. 核心接口 / 页面 / 命令
12. 开发顺序
13. 测试和验收标准
14. 风险和边界
15. 用户审查记录

后续如果方案有问题，请指出具体章节或具体内容，我将直接修改该方案文件。
```

禁止在聊天中重复输出整份方案，除非用户明确要求。

------

## 方案修改规则

如果用户指出方案存在问题，AI 不应重新在聊天中输出一整份新方案。

AI 必须：

1. 读取已有方案文件
2. 根据用户指出的问题定位对应章节
3. 修改方案文件中的对应内容
4. 保持无关章节不变
5. 更新方案文件中的「用户审查记录」
6. 在聊天中只说明修改了哪些章节

回复格式：

```
已修改方案文件：

docs/ai-plans/new-project/<project_name>.md

本次修改：

1. 修改了「第 X 节：...」
   - 原问题：
   - 修改结果：

2. 修改了「第 Y 节：...」
   - 原问题：
   - 修改结果：
```

不要重复输出完整方案，除非用户明确要求。

------

## 方案文件作为后续实现依据

后续如果进入代码实现阶段，代码实现流程应优先读取该方案文件。

代码实现时不应主要依赖聊天上下文中的旧描述，而应以方案文件为准。

如果聊天内容和方案文件冲突，以方案文件为准。

如果用户在聊天中提出新的修改意见，应先更新方案文件，再进入代码实现。

------

## 与功能级方案的关系

本 reference 创建的是项目级方案，不是单个功能的详细实现方案。

项目级方案应该说明：

- 系统整体怎么成立
- 第一版需要哪些核心模块
- 模块之间如何协作
- 第一版怎么验收

项目级方案不应该过度展开：

- 每个函数的详细输入输出
- 每个文件的完整代码设计
- 单个小功能的逐层数据流
- 具体实现代码

如果后续需要对某个模块或功能做细化，应再进入对应的功能级方案或代码实现流程。

------

## 完成标准

本 reference 的任务完成标准是：

1. 已执行需求明确性检查
2. 已确认这是新项目，而不是已有项目继续开发
3. 已明确项目目标和 MVP 范围
4. 已使用 `assets/new_project_plan_template.md`
5. 已创建或更新独立方案文件
6. 已给出方案文件路径
7. 已明确第一版开发顺序
8. 已明确测试和验收标准
9. 没有直接编写代码
10. 没有在聊天中重复输出整份方案

只有方案文件创建或更新完成，才算本 reference 的任务完成。
