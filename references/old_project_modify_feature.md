# 2.2 老项目：修改功能方案 Reference

## 适用场景

当用户希望在一个已经存在的项目中修改某个已有功能的行为、流程、接口、数据结构、状态流转或执行方式时，使用本 reference。

核心要求：

- 项目中已经存在该功能
- 本次需求会改变该功能的行为或实现方式
- 需要分析旧行为与新行为的差异
- 需要评估兼容性、影响范围和迁移需求
- 需要在修改前形成独立方案文件

只要需求本质上是改变已有功能的行为，而不是新增一个原本不存在的能力、修复错误或重构优化，都属于本 reference 的适用范围。

本 reference 只负责 **修改功能前的方案设计**，不负责代码实现。

只要需求本质上是 **改变已有功能的行为**，而不是新增一个原本不存在的能力、修复错误或重构优化，都属于本 reference 的适用范围。

本 reference 只负责 **修改功能前的方案设计**，不负责代码实现。

## 不适用场景与转入规则

如果在需求明确性检查或项目上下文阅读后，发现用户需求不属于“修改功能”，必须停止使用本 reference，并转入对应 reference。

转入规则：

- 如果用户要新增一个原本不存在的功能能力，应转入 本 skill 根目录的 `references/old_project_add_feature.md`
- 如果用户要修复已有错误、异常或不符合预期的行为，应转入 本 skill 根目录的 `references/old_project_fix_bug.md`
- 如果用户只是希望在功能行为不变的前提下优化代码结构、性能、可读性或可维护性，应转入  本 skill 根目录的`references/old_project_refactor_optimize.md`

判断后不要继续创建修改功能方案文件。

应先在聊天中说明转入原因，例如：

```
当前需求不属于修改功能，因为用户不是要改变已有功能行为，而是要修复一个不符合预期的错误。
因此应转入 references/old_project_fix_bug.md。
```

------

## 核心目标

本 reference 的目标不是在聊天中输出完整方案，而是在项目中创建或更新一个独立的 Markdown 方案文件。

该方案文件用于：

- 让用户审查旧功能如何被修改
- 明确旧行为、新行为和差异
- 明确兼容策略、迁移策略和影响范围
- 避免多轮聊天污染上下文
- 让后续代码实现流程只读取方案文件
- 保持“方案设计”和“代码实现”分离

修改功能时，AI 必须先证明自己理解旧行为，再设计新行为。

------

## 重要限制

使用本 reference 时，AI 不允许直接写代码。

禁止行为：

- 不要输出完整代码实现
- 不要生成 patch
- 不要直接进入代码实现
- 不要把方案设计和代码实现混在一起
- 不要在聊天中反复输出整份方案
- 不要跳过旧功能现状分析
- 不要在未理解旧行为前设计新行为
- 不要在未阅读相关配置前判断“不需要配置变化”
- 不要顺手重构无关代码
- 不要引入未说明的新依赖
- 不要忽略兼容旧数据、旧接口、旧调用方
- 不要只说明新流程，不说明旧流程和差异
- 不要只写正常路径，不写异常和边界情况

如果用户要求“直接写代码”，也应先创建或更新方案文件，并说明：

```
当前阶段先创建修改功能方案文件，代码实现应在方案确认后由代码实现流程处理。
```

------

## 当前重点

本 reference 关注的是：

> 项目中已经存在某个功能，现在需要修改该功能的行为、流程、接口、数据结构、状态流转或执行方式。

在方案设计过程中，重点需要回答以下问题：

- 当前旧功能是如何工作的
- 用户希望修改成什么样
- 新旧行为有哪些差异
- 哪些旧逻辑会受到影响
- 是否需要兼容旧接口、旧数据或旧调用方
- 是否需要数据迁移
- 如何在最小改动的前提下完成修改
- 如何验证修改后不会破坏已有功能

所有方案设计都应围绕上述问题展开。

------

## 工作流程

### 第零步：需求明确性检查

在执行本 reference 前，必须先读取并执行本 skill 根目录的`references/common_requirement_clarification.md`，完成需求明确性检查。

只有当用户目标已经明确后，才允许继续本 reference 的后续流程。

如果用户目标不明确，不要读取项目、不要创建方案文件、不要输出完整方案，应先向用户询问最小必要信息。

### 第一步：主动收集上下文

在创建方案文件前，必须先获取并阅读与目标功能相关的项目背景信息。

优先阅读项目内容，而不是直接设计方案或向用户提问。

修改功能必须重点阅读旧功能实现。

需要尽可能阅读和分析：

- 项目整体目录结构
- 项目技术栈
- 项目入口文件
- 旧功能入口
- 旧功能调用链
- 旧接口定义
- 旧接口请求结构
- 旧接口响应结构
- 旧数据模型
- 旧数据库字段
- 旧状态枚举
- 旧业务流程
- 旧异常处理逻辑
- 旧权限逻辑
- 旧任务流程
- 旧前端调用位置
- 旧测试用例
- 旧文档或 README 中对该功能的描述
- 相关配置文件
- 依赖声明文件
- 环境变量定义
- 路由配置
- 权限配置
- 中间件配置
- 任务队列配置
- 缓存配置
- 第三方服务配置

如果目标功能可能涉及依赖、扩展包、环境变量、路由、权限、中间件、任务队列、缓存或第三方服务，必须在方案设计前优先阅读相关配置。

不要等到方案末尾才判断配置变化。

------

### 第二步：判断信息是否足够

如果信息不足，应要求用户提供必要内容。

回复格式：

```
在创建修改功能方案前，我还需要阅读以下内容：

1. ...
2. ...
3. ...

请提供相关文件或目录结构后再继续。
```

如果信息足够，应先简要说明已阅读内容，然后创建方案文件。

回复格式：

```
已阅读的信息：

1. ...
2. ...
3. ...

当前对旧功能的理解：

- ...
- ...
- ...

下面将创建修改功能方案文件。
```

不要在缺少旧功能核心实现的情况下直接设计新方案。

------

### 第三步：创建方案文件

AI 必须根据 `assets/modify_feature_plan_template.md` 创建独立方案文件。

推荐存放路径：

```
docs/ai-plans/modify-feature/<feature_name>.md
```

如果项目中没有该目录，应创建：

```
docs/ai-plans/modify-feature/
```

文件命名规则：

```
<功能名称的英文或拼音简写>.md
```

示例：

```
docs/ai-plans/modify-feature/multi_course_task.md
docs/ai-plans/modify-feature/async_task_execution.md
docs/ai-plans/modify-feature/retry_failed_task.md
docs/ai-plans/modify-feature/task_status_flow.md
```

创建方案文件时：

- 必须使用 `assets/modify_feature_plan_template.md`
- 必须填充模板内容
- 必须基于已阅读的项目上下文填写
- 必须明确旧行为、新行为和差异
- 必须明确兼容策略
- 必须明确是否需要数据迁移
- 必须明确旧行为依赖点
- 不要在聊天中完整复制方案正文
- 不要写代码

------

### 第四步：创建完成后的聊天回复

创建方案文件后，聊天中只需要简要说明文件位置和内容概要。

回复格式：

```
已创建修改功能方案文件：

docs/ai-plans/modify-feature/<feature_name>.md

该文件包含：

1. 修改目标
2. 旧功能现状分析
3. 新功能目标行为
4. 新旧行为对比
5. 兼容策略
6. 文件变更计划
7. 依赖、扩展包和配置变化
8. 最小修改原则
9. 函数设计和调用链
10. 数据结构变化
11. 数据迁移策略
12. 简单用例和完整数据流
13. 异常和边界情况
14. 旧行为依赖点扫描
15. 测试和验收方案
16. 风险和回滚思路
17. 用户审查记录

后续如果方案有问题，请指出具体章节或具体内容，我将直接修改该方案文件。
```

禁止在聊天中重复输出整份方案，除非用户明确要求。

------

## 方案修改规则

如果用户指出方案存在问题，AI 不应重新在聊天中输出一整份新方案。

AI 必须：

1. 读取已有方案文件
2. 根据用户指出的问题定位对应章节
3. 修改方案文件中的对应内容
4. 保持无关章节不变
5. 更新方案文件中的「用户审查记录」
6. 在聊天中只说明修改了哪些章节

回复格式：

```
已修改方案文件：

docs/ai-plans/modify-feature/<feature_name>.md

本次修改：

1. 修改了「第 X 节：...」
   - 原问题：
   - 修改结果：

2. 修改了「第 Y 节：...」
   - 原问题：
   - 修改结果：
```

不要重复输出完整方案，除非用户明确要求。

------

## 方案文件作为后续实现依据

后续如果进入代码实现阶段，代码实现流程应优先读取该方案文件。

代码实现时不应主要依赖聊天上下文中的旧方案描述，而应以方案文件为准。

如果聊天内容和方案文件冲突，以方案文件为准。

如果用户在聊天中提出新的修改意见，应先更新方案文件，再进入代码实现。

------

## 完成标准

本 reference 的任务完成标准是：

1. 已阅读必要项目上下文
2. 已理解旧功能现状
3. 已使用 `assets/modify_feature_plan_template.md`
4. 已创建或更新独立方案文件
5. 已给出方案文件路径
6. 已明确旧行为、新行为和差异
7. 已明确兼容策略和影响范围
8. 没有直接编写代码
9. 没有在聊天中重复输出整份方案

只有方案文件创建或更新完成，才算本 reference 的任务完成。
