# 2.4 老项目：重构优化方案 Reference

## 适用场景

当用户希望在一个已经存在的项目中，在 **不改变功能行为** 的前提下，优化代码结构、可读性、可维护性、性能、模块边界、异常处理、测试结构或工程质量时，使用本 reference。

核心要求：

- 项目中已经存在相关代码或功能
- 当前功能行为原则上是正确的
- 本次目标不是新增功能
- 本次目标不是修改业务行为
- 本次目标不是修复错误行为
- 本次目标是在行为不变的前提下改善实现质量

典型场景：

- 函数太长，需要拆分
- 重复代码太多，需要抽公共逻辑
- view / controller 太臃肿，需要抽 service 层
- 模块职责混乱，需要重新划分边界
- 代码命名不清晰，需要改善可读性
- 异常处理分散，需要统一处理
- 数据库查询太多，需要优化查询
- 接口功能正确，但响应太慢，需要性能优化
- 测试代码重复，需要整理测试结构
- 代码耦合太高，需要降低模块依赖
- 数据结构表达混乱，需要统一内部结构
- 日志记录混乱，需要统一日志方案

只要需求本质上是 **功能行为不变，只改善实现质量**，都属于本 reference 的适用范围。

本 reference 只负责 **重构优化前的方案设计**，不负责代码实现。

------

## 不适用场景与转入规则

如果在需求明确性检查或项目上下文阅读后，发现用户需求不属于“重构优化”，必须停止使用本 reference，并转入对应 reference。

转入规则：

- 如果用户要新增原本不存在的功能，应转入本skill对应根目录的 `references/old_project_add_feature.md`
- 如果用户要改变已有功能行为，应转入本skill对应根目录的  `references/old_project_modify_feature.md`
- 如果用户要修复错误、异常或不符合预期的行为，应转入本skill对应根目录的  `references/old_project_fix_bug.md`

特别注意：

重构优化必须满足：

```
功能行为不变，只改善实现。
```

如果方案中出现以下情况，就不再属于重构优化：

- 改变接口请求格式
- 改变接口响应格式
- 改变业务规则
- 改变状态流转
- 改变权限判断
- 改变用户可见行为
- 修复原本错误的行为
- 新增业务能力
- 改变历史数据含义
- 改变错误码含义
- 改变外部调用方依赖的行为

这些情况应转入对应 reference。

判断后不要继续创建重构优化方案文件。

应先在聊天中说明转入原因，例如：

```
当前需求不属于重构优化，因为用户希望改变已有功能的业务行为，而不是在行为不变的前提下优化代码。
因此应转入 references/old_project_modify_feature.md。
```

------

## 核心目标

本 reference 的目标不是在聊天中输出完整方案，而是在项目中创建或更新一个独立的 Markdown 重构优化方案文件。

该方案文件用于：

- 明确本次重构优化目标
- 记录当前行为基线
- 明确哪些行为不允许改变
- 明确当前代码存在的问题
- 明确重构优化策略
- 明确文件和函数调整计划
- 明确行为等价验证方式
- 明确性能指标，如果本次涉及性能优化
- 避免多轮聊天污染上下文
- 让后续代码实现流程只读取方案文件
- 保持“方案设计”和“代码实现”分离

重构优化时，AI 必须先定义行为基线，再设计结构变化。

------

## 重要限制

使用本 reference 时，AI 不允许直接写代码。

禁止行为：

- 不要输出完整代码实现
- 不要生成 patch
- 不要直接进入代码修改
- 不要把重构优化方案和代码实现混在一起
- 不要在聊天中反复输出整份方案
- 不要跳过当前行为基线分析
- 不要在未确认行为基线前设计重构方案
- 不要改变业务行为
- 不要改变用户可见行为
- 不要改变接口请求格式
- 不要改变接口响应格式
- 不要改变错误码含义
- 不要改变状态流转含义
- 不要改变权限规则
- 不要改变历史数据含义
- 不要借重构之名新增功能
- 不要借优化之名修复未确认的 Bug
- 不要引入未说明的新依赖

如果用户要求“直接重构代码”或“直接优化代码”，也应先创建或更新重构优化方案文件，并说明：

```
当前阶段先创建重构优化方案文件，代码实现应在方案确认后由代码实现流程处理。
```

------

## 当前重点

本 reference 关注的是：

> 功能行为不变，代码实现变好。

在方案设计过程中，重点需要回答以下问题：

- 当前代码或模块的行为基线是什么
- 哪些输入、输出、副作用、异常行为必须保持不变
- 当前代码存在什么质量问题
- 本次优化目标是什么
- 哪些地方允许调整
- 哪些地方绝对不能改变
- 如何用最小改动完成优化
- 如何验证重构前后行为等价
- 如果是性能优化，如何衡量优化前后的指标
- 如何避免重构过程中引入新 Bug

所有方案设计都应围绕上述问题展开。

------

## 工作流程

### 第零步：需求明确性检查

在执行本 reference 前，必须先读取并执行本 skill 根目录的 `references/common_requirement_clarification.md`，完成需求明确性检查。

只有当重构优化目标足够明确后，才允许继续本 reference 的后续流程。

重构优化至少需要明确：

- 要重构或优化哪个文件、模块、接口、函数或功能
- 优化目标是什么
- 是否要求功能行为完全不变
- 是否允许调整文件结构
- 是否允许引入新依赖
- 如何判断优化完成

如果以上信息不足，不要读取项目、不要创建方案文件、不要输出完整方案，应先向用户询问最小必要信息。

------

### 第一步：主动收集上下文

在创建方案文件前，必须先获取并阅读与目标代码或目标模块相关的项目背景信息。

优先阅读项目内容，而不是直接设计方案或向用户提问。

重构优化必须重点阅读当前实现和当前行为。

需要尽可能阅读和分析：

- 项目整体目录结构
- 项目技术栈
- 项目入口文件
- 目标文件或目标模块
- 目标函数或目标类
- 相关调用方
- 相关被调用方
- 相关接口定义
- 相关数据模型
- 相关状态流转
- 相关权限逻辑
- 相关异常处理
- 相关数据库查询
- 相关缓存逻辑
- 相关任务队列逻辑
- 相关配置文件
- 相关测试用例
- 相关 README 或开发文档
- 已有类似模块的实现风格

如果目标优化可能涉及依赖、扩展包、环境变量、路由、权限、中间件、任务队列、缓存或第三方服务，必须在方案设计前优先阅读相关配置。

不要等到方案末尾才判断配置变化。

------

### 第二步：建立当前行为基线

重构优化前必须先建立当前行为基线。

至少需要确认：

- 当前功能入口
- 当前输入结构
- 当前输出结构
- 当前接口请求格式
- 当前接口响应格式
- 当前数据库读写行为
- 当前状态变化
- 当前权限判断
- 当前异常行为
- 当前错误码和错误信息
- 当前日志行为
- 当前外部服务调用
- 当前用户可见结果
- 当前测试覆盖情况

如果无法建立行为基线，不要创建重构优化方案文件。

回复格式：

```
在创建重构优化方案前，我还需要确认当前行为基线：

1. ...
2. ...
3. ...

请提供相关文件、测试用例、接口示例或运行结果后再继续。
```

------

### 第三步：判断项目信息是否足够

如果信息不足，应要求用户提供必要内容。

回复格式：

```
在创建重构优化方案前，我还需要阅读以下内容：

1. ...
2. ...
3. ...

请提供相关文件或目录结构后再继续。
```

如果信息足够，应先简要说明已阅读内容和当前行为基线，然后创建方案文件。

回复格式：

```
已阅读的信息：

1. ...
2. ...
3. ...

当前行为基线：

- 入口：
- 输入：
- 输出：
- 副作用：
- 异常行为：
- 不能改变的行为：

下面将创建重构优化方案文件。
```

不要在缺少行为基线的情况下直接设计重构优化方案。

------

### 第四步：创建方案文件

AI 必须根据 `assets/refactor_optimize_plan_template.md` 创建独立方案文件。

推荐存放路径：

```
docs/ai-plans/refactor-optimize/<target_name>.md
```

如果项目中没有该目录，应创建：

```
docs/ai-plans/refactor-optimize/
```

文件命名规则：

```
<目标模块或优化目标的英文或拼音简写>.md
```

示例：

```
docs/ai-plans/refactor-optimize/task_service.md
docs/ai-plans/refactor-optimize/order_query_performance.md
docs/ai-plans/refactor-optimize/login_module_structure.md
docs/ai-plans/refactor-optimize/export_excel_readability.md
```

创建方案文件时：

- 必须使用 `assets/refactor_optimize_plan_template.md`
- 必须填充模板内容
- 必须基于已阅读的项目上下文填写
- 必须明确当前行为基线
- 必须明确不允许改变的行为
- 必须明确优化目标
- 必须明确行为等价验证方式
- 如果是性能优化，必须明确性能指标和测量方式
- 不要在聊天中完整复制方案正文
- 不要写代码

------

### 第五步：创建完成后的聊天回复

创建方案文件后，聊天中只需要简要说明文件位置和内容概要。

回复格式：

```
已创建重构优化方案文件：

docs/ai-plans/refactor-optimize/<target_name>.md

该文件包含：

1. 重构优化目标
2. 当前行为基线
3. 不允许改变的行为
4. 允许改变的实现细节
5. 当前问题分析
6. 重构优化策略
7. 文件变更计划
8. 函数与模块调整计划
9. 数据结构和接口影响
10. 性能指标和测量方式
11. 行为等价验证
12. 测试和回归验证
13. 风险和回滚思路
14. 用户审查记录

后续如果方案有问题，请指出具体章节或具体内容，我将直接修改该方案文件。
```

禁止在聊天中重复输出整份方案，除非用户明确要求。

------

## 方案修改规则

如果用户指出方案存在问题，AI 不应重新在聊天中输出一整份新方案。

AI 必须：

1. 读取已有方案文件
2. 根据用户指出的问题定位对应章节
3. 修改方案文件中的对应内容
4. 保持无关章节不变
5. 更新方案文件中的「用户审查记录」
6. 在聊天中只说明修改了哪些章节

回复格式：

```
已修改方案文件：

docs/ai-plans/refactor-optimize/<target_name>.md

本次修改：

1. 修改了「第 X 节：...」
   - 原问题：
   - 修改结果：

2. 修改了「第 Y 节：...」
   - 原问题：
   - 修改结果：
```

不要重复输出完整方案，除非用户明确要求。

------

## 方案文件作为后续实现依据

后续如果进入代码实现阶段，代码实现流程应优先读取该方案文件。

代码实现时不应主要依赖聊天上下文中的旧描述，而应以方案文件为准。

如果聊天内容和方案文件冲突，以方案文件为准。

如果用户在聊天中提出新的修改意见，应先更新方案文件，再进入代码实现。

------

## 完成标准

本 reference 的任务完成标准是：

1. 已执行需求明确性检查
2. 已阅读必要项目上下文
3. 已建立当前行为基线
4. 已确认不允许改变的行为
5. 已使用 `assets/refactor_optimize_plan_template.md`
6. 已创建或更新独立方案文件
7. 已给出方案文件路径
8. 已明确行为等价验证方式
9. 如果涉及性能优化，已明确性能指标和测量方式
10. 没有直接编写代码
11. 没有在聊天中重复输出整份方案

只有方案文件创建或更新完成，才算本 reference 的任务完成。
