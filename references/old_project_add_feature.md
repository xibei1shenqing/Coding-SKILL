# 2.1 老项目：添加功能方案 Reference

## 适用场景

当用户希望在一个已经存在的项目中新增某个原本没有的功能时，使用本 reference。

典型场景：

- 项目中新增原本不存在的功能能力
- 在现有业务流程中增加新的业务功能
- 为现有系统扩展新的模块或功能入口
- 新增需要独立设计和实现的业务需求

只要需求本质上是在现有项目中增加新的功能，而不是修改已有功能行为、修复错误或进行重构优化，都属于本 reference 的适用范围。

本 reference 只负责 **添加功能前的方案设计**，不负责代码实现。

## 不适用场景与转入规则

如果在需求明确性检查或项目上下文阅读后，发现用户需求不属于“添加功能”，必须停止使用本 reference，并转入对应 reference。

转入规则：

- 如果用户不是新增原本不存在的功能，而是要改变已有功能的行为，应转入 本 skill 根目录的`references/old_project_modify_feature.md`
- 如果用户不是新增功能，而是要修复已有错误、异常或不符合预期的行为，应转入 本 skill 根目录的`references/old_project_fix_bug.md`
- 如果用户不是新增功能，而是希望在功能行为不变的前提下优化代码结构、性能、可读性或可维护性，应转入本 skill 根目录的 `references/old_project_refactor_optimize.md`

判断后不要继续创建添加功能方案文件。

应先在聊天中说明转入原因，例如：

```
当前需求不属于添加功能，因为项目中已经存在该能力，用户希望改变其行为。
因此应转入 references/old_project_modify_feature.md。
```

------

## 核心目标

本 reference 的目标不是在聊天中输出完整方案，而是在项目中创建或更新一个独立的 Markdown 方案文件。

该方案文件用于：

- 让用户审查功能设计
- 避免多轮聊天污染上下文
- 让后续代码实现流程只读取方案文件
- 保持“方案设计”和“代码实现”分离

------

## 重要限制

使用本 reference 时，AI 不允许直接写代码。

禁止行为：

- 不要输出完整代码实现
- 不要生成 patch
- 不要直接进入代码实现
- 不要把方案设计和代码实现混在一起
- 不要在聊天中反复输出整份方案
- 不要跳过项目上下文阅读
- 不要在未阅读相关配置前判断“不需要配置变化”
- 不要顺手重构无关代码
- 不要引入未说明的新依赖

如果用户要求“直接写代码”，也应先创建或更新方案文件，并说明：

```
当前阶段先创建添加功能方案文件，代码实现应在方案确认后由代码实现流程处理。
```

------

## 工作流程

### 第零步：需求明确性检查

在执行本 reference 前，必须先读取并执行本 skill 根目录的`references/common_requirement_clarification.md`，完成需求明确性检查。

只有当用户目标已经明确后，才允许继续本 reference 的后续流程。

如果用户目标不明确，不要读取项目、不要创建方案文件、不要输出完整方案，应先向用户询问最小必要信息。

### 第一步：主动收集上下文

在创建方案文件前，必须先获取并阅读与目标功能相关的项目背景信息。

优先阅读项目内容，而不是直接设计方案或向用户提问。

需要尽可能阅读和分析：

- 项目整体目录结构
- 项目技术栈
- 项目入口文件
- 与目标功能相关的代码文件
- 相关数据模型
- 相关接口定义
- 相关测试代码
- 已有类似功能的实现方式
- 相关配置文件
- 依赖声明文件
- 环境变量定义
- 路由配置
- 权限配置
- 中间件配置
- 任务队列配置
- 缓存配置
- 第三方服务配置

如果目标功能可能涉及依赖、扩展包、环境变量、路由、权限、中间件、任务队列、缓存或第三方服务，必须在方案设计前优先阅读相关配置。

不要等到方案末尾才判断配置变化。

------

### 第二步：判断信息是否足够

如果信息不足，应要求用户提供必要内容。

回复格式：

```
在创建添加功能方案前，我还需要阅读以下内容：

1. ...
2. ...
3. ...

请提供相关文件或目录结构后再继续。
```

如果信息足够，应先简要说明已阅读内容，然后创建方案文件。

回复格式：

```
已阅读的信息：

1. ...
2. ...
3. ...

当前理解：

- ...
- ...
- ...

下面将创建添加功能方案文件。
```

------

### 第三步：创建方案文件

AI 必须根据 `assets/add_feature_plan_template.md` 创建独立方案文件。

推荐存放路径：

```
docs/ai-plans/add-feature/<feature_name>.md
```

如果项目中没有该目录，应创建：

```
docs/ai-plans/add-feature/
```

文件命名规则：

```
<功能名称的英文或拼音简写>.md
```

示例：

```
docs/ai-plans/add-feature/pause_task.md
docs/ai-plans/add-feature/export_excel.md
docs/ai-plans/add-feature/user_notification.md
```

创建方案文件时：

- 必须使用 `assets/add_feature_plan_template.md`
- 必须填充模板内容
- 必须基于已阅读的项目上下文填写
- 不要在聊天中完整复制方案正文
- 不要写代码

------

### 第四步：创建完成后的聊天回复

创建方案文件后，聊天中只需要简要说明文件位置和内容概要。

回复格式：

```
已创建添加功能方案文件：

docs/ai-plans/add-feature/<feature_name>.md

该文件包含：

1. 功能边界
2. 文件变更计划
3. 依赖、扩展包和配置变化
4. 最小修改原则
5. 函数设计和调用链
6. 数据结构说明
7. 简单用例和完整数据流
8. 异常和边界情况
9. 对旧功能的影响
10. 测试和验收方案
11. 风险和回滚思路
12. 用户审查记录

后续如果方案有问题，请指出具体章节或具体内容，我将直接修改该方案文件。
```

禁止在聊天中重复输出整份方案，除非用户明确要求。

------

## 方案修改规则

如果用户指出方案存在问题，AI 不应重新在聊天中输出一整份新方案。

AI 必须：

1. 读取已有方案文件
2. 根据用户指出的问题定位对应章节
3. 修改方案文件中的对应内容
4. 保持无关章节不变
5. 更新方案文件中的「用户审查记录」
6. 在聊天中只说明修改了哪些章节

回复格式：

```
已修改方案文件：

docs/ai-plans/add-feature/<feature_name>.md

本次修改：

1. 修改了「第 X 节：...」
   - 原问题：
   - 修改结果：

2. 修改了「第 Y 节：...」
   - 原问题：
   - 修改结果：
```

不要重复输出完整方案，除非用户明确要求。

------

## 方案文件作为后续实现依据

后续如果进入代码实现阶段，代码实现流程应优先读取该方案文件。

代码实现时不应主要依赖聊天上下文中的旧方案描述，而应以方案文件为准。

如果聊天内容和方案文件冲突，以方案文件为准。

如果用户在聊天中提出新的修改意见，应先更新方案文件，再进入代码实现。

------

## 完成标准

本 reference 的任务完成标准是：

1. 已阅读必要项目上下文
2. 已使用 `assets/add_feature_plan_template.md`
3. 已创建或更新独立方案文件
4. 已给出方案文件路径
5. 没有直接编写代码
6. 没有在聊天中重复输出整份方案

只有方案文件创建或更新完成，才算本 reference 的任务完成。
