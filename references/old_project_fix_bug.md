# 2.3 老项目：修复 Bug 方案 Reference

## 适用场景

当用户希望修复一个已经存在项目中的错误、异常、不符合预期的行为时，使用本 reference。

核心问题：

当用户希望修复项目中已经存在的错误、异常或不符合预期的行为时，使用本 reference。

本 reference 关注的是：

- 程序无法正常运行
- 程序可以运行但结果不符合预期
- 已有功能出现错误行为
- 已有功能未按预期完成业务流程

本 reference 只负责 **Bug 修复前的诊断和修复方案设计**，不负责代码实现。

## 不适用场景与转入规则

如果在需求明确性检查、Bug 类型判断或项目上下文阅读后，发现用户需求不属于“修复 Bug”，必须停止使用本 reference，并转入对应 reference。

转入规则：

- 如果用户要新增项目中原本不存在的功能，应转入本skill对应根目录的 `references/old_project_add_feature.md`
- 如果用户的需求不是修复错误，而是改变一个已有功能的正常行为，应转入本skill对应根目录的 `references/old_project_modify_feature.md`
- 如果用户的目标是在功能行为不变的前提下优化代码结构、性能、可读性或可维护性，应转入本skill对应根目录的  `references/old_project_refactor_optimize.md`

特别注意：

如果用户说“有 Bug”，但实际含义是“当前功能正常，只是我现在想让它变成另一种行为”，这不是 Bug 修复，应转入 `references/old_project_modify_feature.md`。

如果用户说“优化一下”，但实际目标是修复错误行为，应继续使用本 reference；如果目标只是让代码更清晰或更快，且不改变功能行为，应转入 `references/old_project_refactor_optimize.md`。

判断后不要继续创建 Bug 修复方案文件。

应先在聊天中说明转入原因，例如：

```
当前需求不属于 Bug 修复，因为现有行为并没有被证明是错误行为，用户实际是在提出新的行为要求。
因此应转入 references/old_project_modify_feature.md。
```

------

## 核心目标

本 reference 的目标不是直接修代码，而是在项目中创建或更新一个独立的 Markdown Bug 修复方案文件。

该方案文件用于：

- 记录 Bug 现象
- 记录复现方式
- 判断 Bug 类型
- 分析已确认事实和待确认假设
- 判断问题原因
- 设计修复方案
- 设计验证方式
- 避免多轮聊天污染上下文
- 让后续代码实现流程只读取方案文件

------

## 重要限制

使用本 reference 时，AI 不允许直接写代码。

禁止行为：

- 不要输出完整代码实现
- 不要生成 patch
- 不要直接进入代码修复
- 不要跳过 Bug 复现和诊断
- 不要把 Bug 修复方案和代码实现混在一起
- 不要在聊天中反复输出整份方案
- 不要在没有明确 Bug 现象时直接创建方案文件
- 不要把用户的新需求误判为 Bug
- 不要把无法确认预期行为的情况直接当作 Bug
- 不要基于弱推断直接修改功能行为
- 不要顺手重构无关代码
- 不要引入未说明的新依赖

如果用户要求“直接修代码”，也应先创建或更新 Bug 修复方案文件，并说明：

```
当前阶段先创建 Bug 修复方案文件，代码实现应在方案确认后由代码实现流程处理。
```

------

## 工作流程

### 第零步：需求明确性检查

在执行本 reference 前，必须先读取并执行本 skill 根目录的`references/common_requirement_clarification.md`，完成需求明确性检查。

只有当 Bug 信息足够明确后，才允许继续本 reference 的后续流程。

修复 Bug 至少需要明确：

- Bug 现象是什么
- 如何复现
- 期望结果是什么
- 实际结果是什么
- 是否有报错日志
- 程序是不能运行，还是功能行为不符合预期

如果以上信息不足，不要读取项目、不要创建方案文件、不要输出完整方案，应先向用户询问最小必要信息。

------

## 第一步：判断 Bug 类型

需求明确后，先判断 Bug 类型。

### 类型 A：运行类 Bug

运行类 Bug 指程序或某个执行路径无法正常运行完成。

典型表现：

- 程序无法启动
- 构建失败
- 测试失败
- 命令执行失败
- 接口 500
- 页面运行时报错
- 任务执行过程中断
- 依赖缺失
- 配置缺失
- 数据库连接失败
- 语法错误
- 类型错误
- 运行时异常

如果 Bug 主要表现为“程序跑不起来”或“执行路径中断”，使用：

```
assets/fix_runtime_bug_template.md
```

生成方案文件路径：

```
docs/ai-plans/fix-bug/runtime/<bug_name>.md
```

------

### 类型 B：行为类 Bug

行为类 Bug 指程序可以运行，但结果和预期不一致。

典型表现：

- 接口返回成功，但数据没有保存
- 页面能打开，但显示错误数据
- 状态流转不符合预期
- 任务执行完成，但结果错误
- 导出文件格式不对
- 权限判断结果不对
- 计算结果不对
- 数据更新到了错误位置
- 返回结构不符合预期

如果 Bug 主要表现为“程序能跑，但结果不对”，使用：

```
assets/fix_behavior_bug_template.md
```

生成方案文件路径：

```
docs/ai-plans/fix-bug/behavior/<bug_name>.md
```

------

### 类型 C：两者都有

如果程序某个功能路径运行时报错，导致功能无法完成，优先归为运行类 Bug。

处理顺序：

1. 先创建运行类 Bug 修复方案
2. 先解决运行中断问题
3. 程序能跑通后，再判断是否还存在行为类 Bug

------

## 第二步：行为类 Bug 必须确认预期行为来源

行为类 Bug 的难点是：AI 不一定知道原项目到底应该实现什么功能。

因此，行为类 Bug 必须先确认“预期行为”的依据。

预期行为来源优先级如下：

1. 用户明确描述
2. 相关功能方案文件
3. 已有测试用例
4. README / API 文档 / 使用文档
5. 代码注释 / 类型定义 / 函数命名 / 接口命名
6. 代码上下文推断

如果预期行为来自第 1 至第 5 类，可以创建行为类 Bug 修复方案。

如果预期行为只能来自第 6 或第 7 类，必须标记为弱证据。

弱证据情况下，不应直接进入修复方案，应先向用户确认预期行为。

推荐回复：

```
当前只能根据代码命名、注释或上下文推测这是 Bug，但无法确认真实预期行为。

我需要你确认：

1. 这个功能的正确预期结果是什么？
2. 当前实际结果哪里不符合预期？
3. 是否有 README、测试用例或方案文件可以作为依据？

确认后我再创建行为类 Bug 修复方案文件。
```

------

## 第三步：主动收集上下文

在创建方案文件前，必须阅读与 Bug 相关的项目背景信息。

运行类 Bug 优先阅读：

- 报错日志
- 堆栈信息
- 启动命令
- 测试命令
- 构建命令
- 依赖声明文件
- 配置文件
- 环境变量定义
- 入口文件
- 报错相关代码文件
- 最近修改的文件

行为类 Bug 优先阅读：

- 用户描述的功能入口
- 相关 README 或文档
- 相关方案文件
- 相关测试用例
- 相关接口定义
- 相关业务代码
- 相关数据模型
- 相关前端调用位置
- 相关状态流转
- 相关权限逻辑
- 相关配置文件

如果目标 Bug 可能涉及依赖、扩展包、环境变量、路由、权限、中间件、任务队列、缓存或第三方服务，必须优先阅读相关配置。

不要等到方案末尾才判断配置变化。

------

## 第四步：判断项目信息是否足够

如果信息不足，应要求用户提供必要内容。

运行类 Bug 信息不足时，回复格式：

```
在创建运行类 Bug 修复方案前，我还需要阅读以下内容：

1. 完整报错日志或堆栈
2. 触发报错的命令或操作
3. 相关配置文件或依赖文件
4. 报错涉及的代码文件

请提供相关内容后再继续。
```

行为类 Bug 信息不足时，回复格式：

```
在创建行为类 Bug 修复方案前，我还需要确认以下内容：

1. 功能的预期行为依据
2. 复现步骤
3. 实际结果
4. 相关 README、测试、方案文件或代码文件

请提供相关内容后再继续。
```

如果信息足够，应先简要说明已阅读内容，然后创建方案文件。

运行类 Bug 回复格式：

```
已阅读的信息：

1. ...
2. ...
3. ...

当前判断：

- Bug 类型：运行类 Bug
- 触发方式：
- 直接错误表现：
- 初步影响范围：

下面将创建运行类 Bug 修复方案文件。
```

行为类 Bug 回复格式：

```
已阅读的信息：

1. ...
2. ...
3. ...

当前判断：

- Bug 类型：行为类 Bug
- 预期行为来源：
- 实际行为：
- 差异：
- 初步影响范围：

下面将创建行为类 Bug 修复方案文件。
```

## 方案修改规则

如果用户指出方案存在问题，AI 不应重新在聊天中输出一整份新方案。

AI 必须：

1. 读取已有方案文件
2. 根据用户指出的问题定位对应章节
3. 修改方案文件中的对应内容
4. 保持无关章节不变
5. 更新方案文件中的「用户审查记录」
6. 在聊天中只说明修改了哪些章节

回复格式：

```
已修改方案文件：

docs/ai-plans/fix-bug/<runtime-or-behavior>/<bug_name>.md

本次修改：

1. 修改了「第 X 节：...」
   - 原问题：
   - 修改结果：

2. 修改了「第 Y 节：...」
   - 原问题：
   - 修改结果：
```

不要重复输出完整方案，除非用户明确要求。

------

## 完成标准

本 reference 的任务完成标准是：

1. 已执行需求明确性检查
2. 已判断 Bug 类型
3. 已阅读必要项目上下文
4. 已确认运行类 Bug 的报错依据，或行为类 Bug 的预期行为来源
5. 已使用正确的 assets 模板
6. 已创建或更新独立方案文件
7. 已给出方案文件路径
8. 没有直接编写代码
9. 没有在聊天中重复输出整份方案

只有方案文件创建或更新完成，才算本 reference 的任务完成。
